# บทที่ 1 บทนำ

## 1.1 ความเป็นมาและความสำคัญของปัญหา

ปัจจุบันเทคโนโลยี **Internet of Things (IoT)** ได้เข้ามามีบทบาทในชีวิตประจำวันมากขึ้น โดยเฉพาะในบริบทของ
**บ้านอัจฉริยะ (Smart Home)** ที่ผู้ใช้สามารถควบคุมเครื่องใช้ไฟฟ้าต่าง ๆ ได้จากระยะไกลผ่านโทรศัพท์มือถือ
หรือสั่งงานด้วยเสียง อุปกรณ์ควบคุมแสงสว่าง (หลอดไฟ) เป็นหนึ่งในอุปกรณ์ที่ได้รับความนิยมสูง เนื่องจาก
ใช้งานบ่อยและให้ประโยชน์ทั้งด้านความสะดวกสบายและการประหยัดพลังงาน

อย่างไรก็ตาม ระบบควบคุมหลอดไฟอัจฉริยะที่มีจำหน่ายในท้องตลาดมักประสบปัญหาสำคัญหลายประการ ดังนี้

1. **การพึ่งพาบริการคลาวด์ของผู้ผลิต (Vendor Cloud Dependency)** — อุปกรณ์ส่วนใหญ่ต้องเชื่อมต่อกับ
   เซิร์ฟเวอร์ของผู้ผลิตในต่างประเทศ ทำให้เกิดความหน่วง (latency) ใช้งานไม่ได้เมื่ออินเทอร์เน็ตขัดข้อง
   และมีความเสี่ยงด้านความเป็นส่วนตัวของข้อมูล

2. **ปัญหาความไม่สอดคล้องของสถานะ (State Inconsistency)** — เมื่อมีช่องทางสั่งงานหลายช่องทาง
   (เช่น แอป การสั่งด้วยเสียง สวิตช์จริง การตั้งเวลา) สถานะของหลอดไฟที่แสดงในแต่ละช่องทางอาจไม่ตรงกัน
   เพราะแต่ละฝ่ายต่างเก็บสถานะของตัวเอง ทำให้ผู้ใช้สับสนว่าไฟเปิดหรือปิดอยู่จริง

3. **ความยุ่งยากในการติดตั้งและจับคู่อุปกรณ์ (Provisioning)** — ผู้ใช้ทั่วไปมักต้องตั้งค่ารหัสอุปกรณ์
   หรือเชื่อมต่อ Wi-Fi ด้วยขั้นตอนที่ซับซ้อน

4. **การจัดการอุปกรณ์ที่หลุดการเชื่อมต่อ (Offline Handling)** — เมื่ออุปกรณ์ขาดการเชื่อมต่อ ระบบจำนวนมาก
   ไม่สามารถแยกแยะได้ว่าอุปกรณ์ "ออฟไลน์จริง" หรือ "เพียงแค่ตอบสนองช้า" ส่งผลให้แสดงสถานะผิดพลาด

จากปัญหาข้างต้น โครงงานนี้จึงนำเสนอการพัฒนาระบบควบคุมหลอดไฟที่ทำงานบนเครือข่ายภายในบ้านเป็นหลัก
โดยใช้โพรโทคอล **MQTT** ซึ่งเป็นมาตรฐานการสื่อสารแบบ publish/subscribe ที่ออกแบบมาสำหรับอุปกรณ์ IoT
ที่มีทรัพยากรจำกัด และวางสถาปัตยกรรมให้ **เซิร์ฟเวอร์เป็นศูนย์กลางความจริงเพียงหนึ่งเดียว (Single Source
of Truth)** เพื่อแก้ปัญหาความไม่สอดคล้องของสถานะอย่างเป็นระบบ กล่าวคือ ทุกการเปลี่ยนแปลงสถานะจะวิ่งผ่าน
เส้นทางเดียว ตัดสินใจที่เซิร์ฟเวอร์ บันทึกลงฐานข้อมูล แล้วกระจายสถานะล่าสุดไปยังผู้ใช้ทุกคนแบบเรียลไทม์

นอกจากนี้ ระบบยังออกแบบกลไกการจับคู่อุปกรณ์อัตโนมัติแบบ **Announce & Claim** ที่ผู้ใช้ไม่ต้องกำหนดรหัส
อุปกรณ์เอง และกลไกตรวจจับสถานะออฟไลน์หลายชั้นเพื่อให้ระบบมีความน่าเชื่อถือสูง

## 1.2 วัตถุประสงค์ของโครงงาน

1. เพื่อออกแบบและพัฒนาระบบควบคุมหลอดไฟอัจฉริยะที่ทำงานบนเครือข่ายภายใน (LAN) โดยไม่พึ่งพาคลาวด์
   ของผู้ผลิตภายนอก
2. เพื่อประยุกต์ใช้โพรโทคอล MQTT ในการสื่อสารระหว่างไมโครคอนโทรลเลอร์กับเซิร์ฟเวอร์
3. เพื่อออกแบบสถาปัตยกรรมแบบ Single Source of Truth ที่แก้ปัญหาความไม่สอดคล้องของสถานะระหว่าง
   ช่องทางสั่งงานหลายช่องทาง (หน้าเว็บ เสียง ตั้งเวลา)
4. เพื่อพัฒนากลไกการจับคู่อุปกรณ์อัตโนมัติและการตรวจจับสถานะออนไลน์/ออฟไลน์ที่มีความน่าเชื่อถือ
5. เพื่อพัฒนาหน้าเว็บที่แสดงสถานะของอุปกรณ์แบบเรียลไทม์และสั่งควบคุมได้

## 1.3 ขอบเขตของโครงงาน

**ขอบเขตด้านฮาร์ดแวร์**
- ใช้ไมโครคอนโทรลเลอร์ ESP8266 (บอร์ด NodeMCU v2) เป็นตัวควบคุม
- ควบคุมหลอดไฟผ่านโมดูลรีเลย์ (relay) ชนิด 2 ช่อง (รองรับการควบคุมไฟ 2 ดวงต่อ 1 อุปกรณ์)
- รองรับการสั่งงานด้วยเสียงผ่านโมดูล VC-02-Kit ผ่านการสื่อสาร UART
- อุปกรณ์ต้องอยู่ในเครือข่าย Wi-Fi เดียวกับเซิร์ฟเวอร์

**ขอบเขตด้านซอฟต์แวร์**
- เซิร์ฟเวอร์พัฒนาด้วย Node.js (Express 5) ใช้ฐานข้อมูล MongoDB และใช้ Mosquitto เป็น MQTT broker
- หน้าเว็บพัฒนาด้วย Vue 3 แสดงผลและสั่งงานแบบเรียลไทม์ผ่าน Server-Sent Events
- รองรับการควบคุมแบบ Manual (สั่งเปิด/ปิดเอง) และ Auto (ตั้งเวลาเปิด-ปิดอัตโนมัติ) รวมถึงการตั้งเวลานับถอยหลังปิด
- ระบบบันทึกประวัติการเปลี่ยนสถานะ (device log) ทุกครั้ง

**ข้อจำกัด**
- ระบบออกแบบสำหรับใช้งานภายในเครือข่ายเดียวกัน ไม่รวมการเข้าถึงจากภายนอกผ่านอินเทอร์เน็ต
  (สามารถต่อยอดได้ในอนาคต)
- ไม่ครอบคลุมการหรี่แสง (dimming) แบบต่อเนื่อง — ควบคุมแบบเปิด/ปิดเป็นหลัก
- ไม่รวมระบบยืนยันตัวตนผู้ใช้ (authentication) แบบหลายผู้ใช้

## 1.4 ประโยชน์ที่คาดว่าจะได้รับ

1. ได้ระบบควบคุมหลอดไฟอัจฉริยะที่ทำงานได้เองภายในบ้านโดยไม่ต้องพึ่งคลาวด์ภายนอก ลดความหน่วง
   และเพิ่มความเป็นส่วนตัว
2. ได้ต้นแบบสถาปัตยกรรม Single Source of Truth ที่แก้ปัญหาความไม่สอดคล้องของสถานะ ซึ่งสามารถนำไป
   ประยุกต์กับอุปกรณ์ IoT ประเภทอื่นได้
3. ได้องค์ความรู้และประสบการณ์ในการบูรณาการเทคโนโลยีหลายด้าน ทั้งระบบฝังตัว (embedded) โพรโทคอล
   การสื่อสาร การพัฒนาเว็บแบบเรียลไทม์ และฐานข้อมูล
4. เป็นแนวทางสำหรับผู้ที่สนใจพัฒนาระบบ IoT ที่เน้นความน่าเชื่อถือและการจัดการสถานะอย่างเป็นระบบ

## 1.5 นิยามศัพท์เฉพาะ

| ศัพท์ | ความหมาย |
|------|----------|
| **MQTT** | Message Queuing Telemetry Transport — โพรโทคอลสื่อสารแบบ publish/subscribe สำหรับ IoT |
| **Broker** | ตัวกลางที่รับและกระจายข้อความ MQTT (ในโครงงานนี้ใช้ Mosquitto) |
| **Single Source of Truth (SSOT)** | หลักการให้มีแหล่งข้อมูลที่ถือว่าถูกต้องเพียงแหล่งเดียว ในที่นี้คือเซิร์ฟเวอร์ |
| **Server-Sent Events (SSE)** | เทคนิคให้เซิร์ฟเวอร์ส่งข้อมูลไปยังหน้าเว็บแบบทางเดียวอย่างต่อเนื่อง |
| **Heartbeat** | สัญญาณที่อุปกรณ์ส่งมาเป็นระยะเพื่อบอกว่ายังทำงานอยู่ |
| **LWT (Last Will Testament)** | ข้อความที่ broker จะส่งแทนอุปกรณ์โดยอัตโนมัติเมื่ออุปกรณ์ขาดการเชื่อมต่อ |
| **Retained Message** | ข้อความ MQTT ที่ broker เก็บค่าล่าสุดไว้ ส่งให้ผู้ subscribe ทันทีที่เชื่อมต่อ |
| **QoS** | Quality of Service — ระดับการรับประกันการส่งข้อความของ MQTT (0, 1, 2) |
| **Pairing** | ขั้นตอนการจับคู่อุปกรณ์ใหม่เข้ากับระบบ |
| **Drift Reconciliation** | กลไกปรับสถานะในฐานข้อมูลให้ตรงกับสถานะจริงของฮาร์ดแวร์ |
