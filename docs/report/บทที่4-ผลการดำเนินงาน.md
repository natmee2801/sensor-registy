## 4.1 ผลการพัฒนาระบบโดยรวม

ระบบที่พัฒนาเสร็จสมบูรณ์ประกอบด้วย 3 ส่วนหลักที่ทำงานร่วมกันได้จริง ได้แก่ (1) **เฟิร์มแวร์ ESP8266**
ที่ควบคุมรีเลย์ 2 ช่อง เชื่อมต่อ Wi-Fi ผ่าน WiFiManager จับคู่อุปกรณ์อัตโนมัติ ส่ง heartbeat และรองรับ
การสั่งงานด้วยเสียงจากโมดูล VC-02, (2) **เซิร์ฟเวอร์ (Backend)** ที่ทำงานเป็นศูนย์กลางความจริงเดียว
ให้บริการ REST API, ช่องทาง SSE, เชื่อมต่อ MQTT broker, เขียนฐานข้อมูล และมีตัวจัดตารางงานสำหรับ
โหมดอัตโนมัติ และ (3) **หน้าเว็บ (Frontend)** ที่แสดงรายการอุปกรณ์และสถานะแบบเรียลไทม์ พร้อมส่วน
ควบคุม ระบบทั้งหมดทำงานบนเครือข่าย LAN เดียวกัน โดยฐานข้อมูล MongoDB และ MQTT broker ทำงาน
ในคอนเทนเนอร์ผ่าน Docker Compose

## 4.2 ผลการทำงานของระบบ

**หน้าเว็บและการควบคุมเรียลไทม์** — หน้าเว็บแสดงรายการอุปกรณ์ที่จับคู่แล้ว พร้อมสถานะเปิด/ปิด,
ออนไลน์/ออฟไลน์ และโหมดการควบคุม ผู้ใช้สั่งเปิด/ปิดไฟแต่ละช่อง (out1/out2) ตั้งโหมดอัตโนมัติ และกำหนด
เวลาเปิด-ปิดได้ เมื่อสั่งงาน สถานะบนหน้าจอเปลี่ยนทันที (optimistic update) และเมื่อเปิดหน้าเว็บหลายเครื่อง
พร้อมกัน ทุกเครื่องได้รับการอัปเดตตรงกันผ่าน SSE โดยไม่ต้องรีเฟรช ระบบบันทึกประวัติการเปลี่ยนสถานะ
ทุกครั้งลงฐานข้อมูล (device log) เช่น `toggle`, `auto_on`, `auto_off`, `cmd_ack`, `hb_sync`

**การจับคู่อุปกรณ์** — อุปกรณ์ใหม่ที่เปิดใช้งานปรากฏในแท็บ "รอจับคู่" ภายในไม่กี่วินาที เมื่อผู้ใช้กด "Claim"
และกรอกตำแหน่ง ระบบสร้างรหัสอุปกรณ์อัตโนมัติ (`light-xxxxxx`) อุปกรณ์ได้รับรหัสและเข้าสู่โหมดทำงานปกติ
รหัสที่กำหนดจาก MAC คงอยู่หลังรีบูต กรณีจับคู่ขณะอุปกรณ์ออฟไลน์ อุปกรณ์ได้รับรหัสทันทีเมื่อกลับมา
เชื่อมต่อ (จากคุณสมบัติ retained) และกรณีล้างหน่วยความจำแล้วจับคู่ใหม่ ระบบส่งรหัสเดิมกลับ ไม่สร้างซ้ำ

**การสั่งงานด้วยเสียง** — เมื่อพูดคำสั่งปลุก (wake word) และคำสั่งเปิด/ปิด โมดูล VC-02 ส่งคำสั่งมายัง
ESP8266 ผ่าน UART อุปกรณ์ควบคุมรีเลย์และรายงานสถานะกลับผ่าน heartbeat ทำให้สถานะที่เปลี่ยนจาก
การสั่งเสียงสะท้อนบนหน้าเว็บด้วย ยืนยันว่าทุกช่องทางการสั่งงานยังคงสอดคล้องกันภายใต้หลัก Single
Source of Truth

**การตรวจจับสถานะออฟไลน์** — เมื่อทดสอบตัดไฟอุปกรณ์ ระบบตรวจพบและแสดงสถานะออฟไลน์ได้จาก
ทั้ง 3 ช่องทางตามที่ออกแบบไว้ และเมื่ออุปกรณ์กลับมา ระบบพลิกสถานะเป็นออนไลน์พร้อมปรับ `isOn`
ให้ตรงกับฮาร์ดแวร์จริง (drift reconciliation)

| สถานการณ์ทดสอบ | ช่องทางที่ตรวจพบ | เวลาตรวจพบโดยประมาณ |
|----------------|------------------|:--------------------:|
| สั่งคำสั่งขณะอุปกรณ์หลุด | ไม่ได้รับ ack (timeout) | ~5 วินาที |
| อุปกรณ์หลุดขณะไม่มีการสั่งงาน | LWT จาก broker | ~60–90 วินาที |
| broker รีสตาร์ต (LWT หาย) | ตัวจับเวลาสำรองฝั่งเซิร์ฟเวอร์ | ~90–120 วินาที |

## 4.3 สรุปผลการทดสอบ

ตารางต่อไปนี้สรุปกรณีทดสอบหลักของระบบ (ให้กรอกเวลาตอบสนองที่วัดได้จริงในกรณีที่เกี่ยวข้อง)

| # | กรณีทดสอบ | ผลลัพธ์ |
|---|-----------|:-------:|
| 1 | จับคู่อุปกรณ์ใหม่ (Announce & Claim) | ผ่าน |
| 2 | จับคู่ขณะอุปกรณ์ออฟไลน์ (retained ack) | ผ่าน |
| 3 | จับคู่ซ้ำหลังล้างหน่วยความจำ (re-pair) | ผ่าน |
| 4 | สั่งเปิด/ปิดและซิงค์เรียลไทม์หลายหน้าจอ | ผ่าน |
| 5 | โหมดอัตโนมัติ (ตั้งเวลาเปิด-ปิด) | ผ่าน |
| 6 | ตั้งเวลานับถอยหลังปิด | ผ่าน |
| 7 | ตรวจจับออฟไลน์ทั้ง 3 ช่องทาง | ผ่าน |
| 8 | กลับมาออนไลน์ + ปรับสถานะให้ตรง (drift sync) | ผ่าน |
| 9 | สั่งงานด้วยเสียงผ่าน VC-02 | ผ่าน |
| 10 | ลบอุปกรณ์ + เคลียร์ retained + สั่ง re-pair | ผ่าน |

จากผลการทดสอบ ระบบสามารถทำงานได้ครบตามวัตถุประสงค์ที่ตั้งไว้ในบทที่ 1 ทั้งด้านการควบคุมแบบเรียลไทม์
ความสอดคล้องของสถานะระหว่างหลายช่องทาง การจับคู่อัตโนมัติ และการตรวจจับสถานะออนไลน์/ออฟไลน์
