# บทที่ 2 ทฤษฎีและงานที่เกี่ยวข้อง

บทนี้กล่าวถึงทฤษฎี เทคโนโลยี และเครื่องมือที่เกี่ยวข้องกับการพัฒนาโครงงาน เพื่อเป็นพื้นฐานความเข้าใจ
ในการออกแบบระบบที่จะนำเสนอในบทที่ 3

## 2.1 Internet of Things และ Smart Home

**Internet of Things (IoT)** หมายถึงการเชื่อมต่ออุปกรณ์ทางกายภาพ (สิ่งของต่าง ๆ) เข้ากับเครือข่าย
เพื่อให้สามารถเก็บข้อมูล สื่อสาร และควบคุมระยะไกลได้ อุปกรณ์ IoT มักมีทรัพยากรจำกัด (หน่วยความจำ
พลังประมวลผล และพลังงานต่ำ) จึงต้องการโพรโทคอลการสื่อสารที่มีน้ำหนักเบา

**Smart Home** เป็นการประยุกต์ใช้ IoT ภายในที่อยู่อาศัย ครอบคลุมการควบคุมแสงสว่าง เครื่องปรับอากาศ
ระบบรักษาความปลอดภัย ฯลฯ สถาปัตยกรรมของ Smart Home แบ่งได้เป็น 2 รูปแบบหลัก คือ

- **Cloud-based** — อุปกรณ์เชื่อมต่อกับเซิร์ฟเวอร์ของผู้ผลิตบนอินเทอร์เน็ต ข้อดีคือเข้าถึงจากที่ใดก็ได้
  แต่มีข้อเสียด้านความหน่วง ความเป็นส่วนตัว และต้องพึ่งพาการเชื่อมต่ออินเทอร์เน็ต
- **Local/Edge-based** — อุปกรณ์และตัวควบคุมทำงานภายในเครือข่ายเดียวกัน ลดความหน่วงและทำงานได้แม้
  อินเทอร์เน็ตขัดข้อง โครงงานนี้เลือกใช้แนวทางนี้

## 2.2 โพรโทคอล MQTT

**MQTT (Message Queuing Telemetry Transport)** เป็นโพรโทคอลการสื่อสารแบบ **publish/subscribe**
ที่ทำงานบน TCP/IP ออกแบบมาให้มีน้ำหนักเบา เหมาะกับอุปกรณ์ IoT องค์ประกอบหลักมีดังนี้

- **Broker** — เซิร์ฟเวอร์ตัวกลางที่รับข้อความจากผู้ส่ง (publisher) แล้วกระจายไปยังผู้รับ (subscriber)
- **Topic** — ชื่อช่องทาง (คล้ายเส้นทางไฟล์ เช่น `dev/light-01/cmd`) ที่ใช้จัดกลุ่มข้อความ ผู้รับสามารถใช้
  สัญลักษณ์ wildcard เช่น `+` (หนึ่งระดับ) หรือ `#` (หลายระดับ) เพื่อ subscribe หลาย topic พร้อมกัน
- **Publisher / Subscriber** — ผู้ส่งและผู้รับข้อความ โดยทั้งสองฝ่ายไม่จำเป็นต้องรู้จักกันโดยตรง สื่อสารผ่าน
  broker เท่านั้น ทำให้ระบบ decoupling ได้ดี

**คุณสมบัติสำคัญของ MQTT ที่โครงงานนำมาใช้:**

1. **Quality of Service (QoS)** — ระดับการรับประกันการส่งข้อความ มี 3 ระดับ
   - QoS 0 (at most once) — ส่งครั้งเดียว ไม่รับประกัน เหมาะกับข้อมูลที่ส่งบ่อยเช่น heartbeat
   - QoS 1 (at least once) — รับประกันว่าถึงผู้รับอย่างน้อยหนึ่งครั้ง (อาจซ้ำ) เหมาะกับคำสั่งควบคุม
   - QoS 2 (exactly once) — รับประกันส่งถึงเพียงครั้งเดียว มี overhead สูงสุด

2. **Retained Message** — เมื่อ publish ด้วยแฟล็ก retained broker จะเก็บข้อความล่าสุดของ topic นั้นไว้
   และส่งให้ผู้ subscribe ทันทีที่เชื่อมต่อ คุณสมบัตินี้มีประโยชน์มากในการเก็บ "สถานะล่าสุด" เช่น heartbeat
   หรือผลการจับคู่อุปกรณ์ ทำให้ผู้รับที่เพิ่งเชื่อมต่อได้ค่าปัจจุบันทันทีโดยไม่ต้องรอรอบถัดไป

3. **Last Will and Testament (LWT)** — ขณะเชื่อมต่อ ผู้ส่งสามารถฝากข้อความ "พินัยกรรม" ไว้กับ broker
   หากผู้ส่งขาดการเชื่อมต่อแบบผิดปกติ (เช่น ไฟดับ สายหลุด) broker จะ publish ข้อความนี้แทนโดยอัตโนมัติ
   คุณสมบัตินี้ทำให้ระบบตรวจจับอุปกรณ์ออฟไลน์ได้โดยไม่ต้องเขียนตัวจับเวลาเองทั้งหมด

ในโครงงานนี้ใช้ **Eclipse Mosquitto** เป็น MQTT broker ซึ่งเป็นซอฟต์แวร์โอเพนซอร์สที่นิยมและรองรับ
คุณสมบัติข้างต้นครบถ้วน

## 2.3 ไมโครคอนโทรลเลอร์ ESP8266 / NodeMCU

**ESP8266** เป็นชิปไมโครคอนโทรลเลอร์ที่มีโมดูล Wi-Fi ในตัว ราคาประหยัด เป็นที่นิยมในงาน IoT
โครงงานนี้ใช้บอร์ด **NodeMCU v2** ซึ่งเป็นบอร์ดพัฒนาที่มี ESP8266 เป็นแกนกลาง พร้อมพอร์ต USB
และวงจรจ่ายไฟ พัฒนาเฟิร์มแวร์ด้วยกรอบงาน **Arduino** บน **PlatformIO** ไลบรารีหลักที่ใช้ ได้แก่

- **PubSubClient** — ไลบรารี MQTT client สำหรับ Arduino/ESP8266
- **ArduinoJson** — ไลบรารีสำหรับสร้างและแปลงข้อมูล JSON
- **WiFiManager** — ไลบรารีจัดการการเชื่อมต่อ Wi-Fi แบบ captive portal เมื่อยังไม่ได้ตั้งค่า อุปกรณ์จะ
  เปิดเป็น Access Point ให้ผู้ใช้เลือกเครือข่ายและกรอกรหัสผ่านผ่านหน้าเว็บ โดยไม่ต้องฮาร์ดโค้ดรหัสผ่าน
- **LittleFS** — ระบบไฟล์ในหน่วยความจำแฟลชของ ESP8266 ใช้เก็บค่ารหัสอุปกรณ์ (device id) ให้คงอยู่
  แม้รีบูต

นอกจากนี้ยังใช้โมดูลเสียง **VC-02-Kit** ที่สื่อสารกับ ESP8266 ผ่าน UART (SoftwareSerial) เพื่อรับคำสั่ง
เสียงแบบออฟไลน์ (offline voice recognition) สำหรับสั่งเปิด/ปิดไฟ

## 2.4 เทคโนโลยีฝั่งเซิร์ฟเวอร์ (Backend)

เซิร์ฟเวอร์พัฒนาด้วย **Node.js** โดยใช้ภาษา **TypeScript** เพื่อความปลอดภัยของชนิดข้อมูล เครื่องมือและ
ไลบรารีหลักประกอบด้วย

- **Express 5** — เฟรมเวิร์กเว็บสำหรับสร้าง REST API และจัดการ HTTP
- **Mongoose 8** — Object Data Modeling (ODM) สำหรับเชื่อมต่อและจัดการข้อมูลใน MongoDB พร้อมการ
  ตรวจสอบความถูกต้องของข้อมูล (schema validation)
- **MongoDB 7** — ฐานข้อมูลแบบ NoSQL (document database) เก็บข้อมูลอุปกรณ์และประวัติการทำงาน
- **mqtt.js** — ไลบรารี MQTT client ฝั่ง Node.js ทำให้เซิร์ฟเวอร์เป็นทั้ง publisher และ subscriber
- **zod** — ไลบรารีตรวจสอบความถูกต้องของข้อมูลขาเข้า (validation) ที่ขอบเขต API
- **node-cron** — ตัวจัดตารางงาน (scheduler) สำหรับงานที่ทำงานตามเวลา เช่น เปิด/ปิดไฟอัตโนมัติ
  และตรวจสอบอุปกรณ์ที่หายไป
- **pino** — ไลบรารีบันทึก log ของเซิร์ฟเวอร์ที่มีประสิทธิภาพสูง

**Server-Sent Events (SSE)** เป็นเทคนิคที่เซิร์ฟเวอร์ส่งข้อมูลไปยังหน้าเว็บแบบทางเดียวอย่างต่อเนื่องผ่าน
การเชื่อมต่อ HTTP ที่เปิดค้างไว้ เหมาะกับการ push การอัปเดตสถานะแบบเรียลไทม์ (เช่น สถานะหลอดไฟ
เปลี่ยน) โดยหน้าเว็บใช้ `EventSource` ในการรับ และมีคุณสมบัติ reconnect อัตโนมัติเมื่อการเชื่อมต่อหลุด
SSE มีน้ำหนักเบากว่า WebSocket และเหมาะกับกรณีที่ข้อมูลไหลทางเดียวจากเซิร์ฟเวอร์สู่ผู้ใช้

## 2.5 เทคโนโลยีฝั่งหน้าเว็บ (Frontend)

หน้าเว็บพัฒนาด้วย **Vue 3** ซึ่งเป็นเฟรมเวิร์ก JavaScript สำหรับสร้าง Single Page Application (SPA)
เครื่องมือที่เกี่ยวข้องประกอบด้วย

- **Pinia** — ไลบรารีจัดการสถานะส่วนกลาง (state management) ของแอป
- **Vue Router** — จัดการเส้นทาง (routing) ภายใน SPA
- **Vite** — เครื่องมือ build และ dev server ที่รวดเร็ว
- **Tailwind CSS** — เฟรมเวิร์ก CSS แบบ utility-first สำหรับจัดรูปแบบหน้าเว็บ

หน้าเว็บรับสถานะอุปกรณ์แบบเรียลไทม์ผ่าน SSE และส่งคำสั่งควบคุมผ่าน REST API โดยยึดหลักว่า
เซิร์ฟเวอร์เป็นผู้ตัดสินสถานะ หน้าเว็บเพียงแสดงผลตามข้อมูลที่ได้รับ

## 2.6 งานและระบบที่เกี่ยวข้อง

ระบบบ้านอัจฉริยะเชิงพาณิชย์ เช่น Philips Hue, TP-Link Tapo หรือ Google Home ส่วนใหญ่ทำงานผ่าน
คลาวด์ของผู้ผลิต ซึ่งให้ความสะดวกในการเข้าถึงจากภายนอก แต่มีข้อจำกัดด้านความเป็นส่วนตัว ความหน่วง
และการพึ่งพาอินเทอร์เน็ต

ในกลุ่มโอเพนซอร์ส **Home Assistant** เป็นแพลตฟอร์มบ้านอัจฉริยะที่ทำงานภายในเครือข่าย (local-first)
และรองรับ MQTT รวมถึงมาตรฐาน **Tasmota / ESPHome** สำหรับเฟิร์มแวร์ ESP8266/ESP32 ซึ่งแสดงให้เห็นว่า
แนวทาง local-first ที่ใช้ MQTT เป็นที่ยอมรับและใช้งานจริง

อย่างไรก็ตาม ระบบเหล่านี้เป็นแพลตฟอร์มขนาดใหญ่ที่รองรับอุปกรณ์หลากหลาย โครงงานนี้แตกต่างตรงที่
มุ่งเน้นการออกแบบสถาปัตยกรรม **Single Source of Truth** และการจัดการ **สถานะ/การจับคู่/การตรวจจับ
ออฟไลน์** อย่างละเอียดสำหรับอุปกรณ์ควบคุมแสงสว่างโดยเฉพาะ เพื่อศึกษาและสาธิตหลักการออกแบบระบบ
แบบกระจาย (distributed system) ที่มีความสอดคล้องของสถานะสูง มากกว่าการเป็นแพลตฟอร์มเอนกประสงค์
