# บทที่ 3 วิธีดำเนินโครงงาน

บทนี้นำเสนอการออกแบบระบบและวิธีดำเนินงาน โดยกล่าวถึงเฉพาะส่วนสำคัญหลักของสถาปัตยกรรม ได้แก่
ภาพรวมระบบ การออกแบบ MQTT topology, และการไหลของเหตุการณ์ตามหลัก Single Source of Truth
ปิดท้ายด้วยเครื่องมือและแผนการดำเนินงาน

## 3.1 ภาพรวมสถาปัตยกรรมระบบ

ระบบประกอบด้วย 5 องค์ประกอบหลักที่สื่อสารกันบนเครือข่ายภายใน (LAN) เดียวกัน

```
┌──────────────┐      MQTT       ┌──────────────┐
│   ESP8266    │◄───────────────►│  Mosquitto   │
│  (NodeMCU)   │   pub/sub 1883  │   (Broker)   │
│  + Relay 2ch │                 └──────┬───────┘
│  + VC-02     │                        │ MQTT
└──────────────┘                        ▼
                                 ┌──────────────┐     ┌──────────────┐
                                 │   Backend    │────►│  MongoDB 7   │
                                 │  Express 5   │     │  (devices,   │
                                 │  + mqtt.js   │     │   logs)      │
                                 └──────┬───────┘     └──────────────┘
                                        │ REST + SSE
                                        ▼
                                 ┌──────────────┐
                                 │  Frontend    │
                                 │   Vue 3 SPA  │
                                 └──────────────┘
```

หน้าที่ของแต่ละองค์ประกอบ

| องค์ประกอบ | หน้าที่ |
|-----------|--------|
| **ESP8266 (NodeMCU)** | ควบคุมรีเลย์เปิด/ปิดไฟ 2 ช่อง รับคำสั่งเสียงจาก VC-02 ส่ง heartbeat และ ack กลับผ่าน MQTT |
| **Mosquitto (Broker)** | ตัวกลางรับ-ส่งข้อความ MQTT พอร์ต 1883 |
| **Backend (Express 5)** | ศูนย์กลางความจริงเดียว — รับคำสั่ง ตัดสินสถานะ บันทึก DB และกระจายผ่าน SSE |
| **MongoDB 7** | เก็บข้อมูลอุปกรณ์ สถานะ และประวัติการทำงาน |
| **Frontend (Vue 3)** | แสดงสถานะเรียลไทม์และส่งคำสั่งควบคุม |

**หลักการสำคัญ:** Backend เป็น **Single Source of Truth** ทุกการเปลี่ยนแปลงสถานะ ไม่ว่าจะมาจากผู้ใช้
(หน้าเว็บ) การสั่งด้วยเสียง การตั้งเวลาอัตโนมัติ หรือสัญญาณจากตัวอุปกรณ์ จะต้องผ่านเซิร์ฟเวอร์เป็นจุดเดียว
แล้วเซิร์ฟเวอร์จึงกระจายสถานะล่าสุดไปยังหน้าเว็บทุกเครื่อง — หน้าเว็บไม่เก็บสถานะของตัวเอง และไม่ poll

## 3.2 การออกแบบ MQTT Topology

ตารางต่อไปนี้สรุป topic ทั้งหมดที่ใช้สื่อสารระหว่าง ESP8266 กับ backend ผ่าน broker

| Topic | ทิศทาง | QoS | Retained | Payload | จุดประสงค์ |
|-------|--------|-----|----------|---------|-----------|
| `pair/hello` | ESP → backend | 1 | ไม่ | `{mac, model, fw}` | อุปกรณ์ที่ยังไม่จับคู่ ประกาศตัวทุก 5 วินาที |
| `pair/ack/{mac}` | backend → ESP | 1 | **ใช่** | `{device_id}` | แจ้งรหัสอุปกรณ์ที่เซิร์ฟเวอร์กำหนดให้ |
| `dev/{id}/cmd` | backend → ESP | 1 | ไม่ | `{cmd_id, action, isOn, ts}` | สั่งเปิด/ปิด |
| `dev/{id}/ack` | ESP → backend | 1 | ไม่ | `{cmd_id, status, isOn}` | ตอบรับคำสั่ง |
| `dev/{id}/hb` | ESP → backend | 0 | ใช่ | `{isOn, rssi, uptime}` | heartbeat ทุก 30 วินาที |
| `dev/{id}/lwt` | broker → backend | 1 | ใช่ | `"offline"` | broker แจ้งเมื่ออุปกรณ์หลุดการเชื่อมต่อ |
| `dev/{id}/wipe` | backend → ESP | 1 | ไม่ | `"1"` | สั่งอุปกรณ์ลืมรหัสเดิมแล้วจับคู่ใหม่ |

**การออกแบบที่สำคัญ**

- **`pair/ack/{mac}` และ `dev/{id}/hb` เป็น retained** — ผู้รับที่เพิ่งเชื่อมต่อจะได้ค่าล่าสุดทันที เช่น อุปกรณ์
  ได้รหัสจับคู่แม้กดตอนออฟไลน์ และเซิร์ฟเวอร์ได้สถานะล่าสุดทันทีหลังรีสตาร์ต
- **`dev/{id}/lwt`** อาศัยกลไก LWT ของ MQTT — อุปกรณ์ฝากไว้กับ broker ว่า "หากฉันหลุดให้ประกาศ
  offline" จึงตรวจจับออฟไลน์ได้อัตโนมัติ
- **คำสั่งควบคุมใช้ QoS 1** เพื่อรับประกันการส่งถึง ส่วน heartbeat ใช้ QoS 0 เพราะส่งบ่อยและยอมรับ
  การสูญหายเป็นครั้งคราวได้

เซิร์ฟเวอร์ subscribe แบบ wildcard (`pair/hello`, `dev/+/ack`, `dev/+/hb`, `dev/+/lwt`) จึงรองรับอุปกรณ์
จำนวนมากด้วยการ subscribe เพียงไม่กี่ topic

## 3.3 Event Flow และหลักการ Single Source of Truth

ทุกการเปลี่ยนแปลงสถานะวิ่งผ่านเส้นทางเดียวกัน ดังแผนภาพ

```
[ ผู้ใช้กดปุ่ม / สั่งเสียง / ตั้งเวลา / สัญญาณ MQTT เข้า ]
        │
        ▼
  Controller  ◄── ตรวจสอบข้อมูลด้วย zod
        │
        ▼
  Service (business logic) ── เขียน MongoDB + บันทึก DeviceLog
        │
        ├─► emit เหตุการณ์เข้า EventEmitter ภายใน
        │
        └─► (ถ้าเป็นคำสั่งเปิด/ปิด) publish คำสั่งผ่าน MQTT แบบ fire-and-forget
        │
        ▼
  SSE Controller  (GET /api/events, text/event-stream)
        │   ส่ง event: device_updated / device_removed / ...
        ▼
  Frontend (EventSource) ──► อัปเดต Pinia store ──► หน้าจอเปลี่ยนทันที
```

**คุณสมบัติเด่นของการออกแบบนี้**

1. **เส้นทางเดียว (single path):** ไม่ว่าคำสั่งจะมาจากที่ใด ทุกอย่างผ่าน Service layer ที่เขียน DB และ emit
   เหตุการณ์ ทำให้สถานะมีความสอดคล้องเสมอ
2. **กระจายแบบ broadcast:** หน้าเว็บทุกเครื่องที่เชื่อมต่อ SSE จะได้รับการอัปเดตเหมือนกัน จึงเห็นสถานะ
   ตรงกัน
3. **Optimistic update ฝั่งเซิร์ฟเวอร์:** เมื่อผู้ใช้สั่งเปิด/ปิด เซิร์ฟเวอร์จะเขียน DB และส่ง SSE ทันที **ก่อน**
   ที่จะรอการตอบรับ (ack) จากอุปกรณ์ ทำให้หน้าจอตอบสนองรวดเร็ว แล้วจึงปรับแก้ภายหลังหากพบความ
   ไม่ตรงกันผ่านสัญญาณ ack และ heartbeat

นอกจากการควบคุมแบบ manual ระบบยังรองรับ **โหมดอัตโนมัติ (auto)** ที่ตั้งเวลาเปิด-ปิดได้ และการตั้งเวลา
นับถอยหลังปิด โดยมีตัวจัดตารางงาน (node-cron) ทำงานเป็นระยะเพื่อตรวจและสั่งงานตามเวลาที่กำหนด ทั้งหมด
ยังคงวิ่งผ่านเส้นทางเดียวกันกับการสั่งด้วยมือ

> **หมายเหตุ:** กลไกระดับรายละเอียด เช่น วงจรชีวิตของคำสั่ง (command lifecycle), การปรับสถานะให้ตรงกัน
> ระหว่างฐานข้อมูลกับฮาร์ดแวร์ (drift reconciliation via heartbeat), ขั้นตอนการจับคู่อุปกรณ์ (Announce &
> Claim), และการตรวจจับสถานะออฟไลน์หลายชั้น จะนำเสนออย่างละเอียดในรายงานฉบับสมบูรณ์ ในที่นี้ขอ
> สรุปแนวคิดหลักไว้ในหัวข้อ 3.4

## 3.4 กลไกความน่าเชื่อถือของระบบ (สรุป)

เพื่อให้ระบบมีความน่าเชื่อถือ ได้ออกแบบกลไกหลัก 2 ส่วน โดยสรุปแนวคิดดังนี้

**การจับคู่อุปกรณ์อัตโนมัติ (Announce & Claim)** — ผู้ใช้ไม่ต้องกำหนดรหัสอุปกรณ์เอง เมื่ออุปกรณ์ใหม่บูต
จะประกาศ MAC ของตัวเองผ่าน `pair/hello` ทุก 5 วินาที หน้าเว็บแสดงเป็นรายการ "รอจับคู่" ผู้ใช้เพียงกด
"Claim" และกรอกตำแหน่งติดตั้ง เซิร์ฟเวอร์จะสร้างรหัสอุปกรณ์อัตโนมัติจาก MAC (รูปแบบ `light-` + 6 ตัวท้าย
ของ MAC) แล้วส่งกลับผ่าน `pair/ack/{mac}` แบบ retained อุปกรณ์บันทึกรหัสลงหน่วยความจำแล้วรีสตาร์ตเข้าสู่
โหมดทำงานปกติ

**การตรวจจับสถานะออฟไลน์** — ระบบตรวจจับว่าอุปกรณ์หลุดการเชื่อมต่อจาก 3 ช่องทางที่เสริมกัน คือ
(1) เมื่อสั่งคำสั่งแล้วไม่ได้รับ ack ภายในเวลาที่กำหนด (เร็วสุด ~5 วินาที), (2) เมื่อ broker ตรวจพบการหลุด
การเชื่อมต่อแล้ว publish ข้อความ LWT โดยอัตโนมัติ และ (3) ตัวจับเวลาฝั่งเซิร์ฟเวอร์ที่ตรวจสอบอุปกรณ์ที่
ขาด heartbeat เป็นระยะ (สำรอง) การมีหลายช่องทางช่วยให้ระบบทนทานต่อความผิดพลาดและรายงานสถานะ
ออนไลน์/ออฟไลน์ได้แม่นยำ
