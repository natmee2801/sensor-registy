# Spec: แผนการเชื่อมต่อระบบ IoT สำหรับหลอดไฟสั่งการด้วยเสียง

**วันที่:** 2026-06-16
**ประเภท:** เอกสาร/แผนผังประกอบรายงาน (ไม่ใช่การเขียนโค้ด)
**สถานะ:** อนุมัติโครงแล้ว รอ review spec

## เป้าหมาย

สร้างเอกสารประกอบรายงานไทยที่อธิบาย **แผนการเชื่อมต่อ (connectivity plan)** ของ path
คำสั่งเสียงในระบบจริง โดยเล่าตามเส้นทางจริงของคำสั่งเสียง 1 คำตั้งแต่ผู้ใช้พูดจนเห็นผล
บนหน้าเว็บ เน้น "เข้าใจง่ายที่สุด ดูระบบง่ายที่สุด" ตามที่ผู้ใช้ระบุ

ระบบที่อธิบายมีอยู่จริงแล้ว (ไม่ต้องสร้างฟีเจอร์ใหม่) — เอกสารนี้เป็นการเรียบเรียง
การเชื่อมต่อที่มีอยู่ให้เป็นภาษารายงาน

## ขอบเขต

- **รวม:** path เสียง end-to-end (เสียง → VC-02 → UART → ESP8266 → WiFi → MQTT →
  Backend → SSE → หน้าเว็บ), พารามิเตอร์การเชื่อมต่อแต่ละช่วง, ลำดับเหตุการณ์, ตารางสรุป
- **ไม่รวม:** path การสั่งจากหน้าเว็บ (`dev/{id}/cmd`), cron/auto schedule, pairing flow
  รายละเอียด (อ้างถึงได้แต่ไม่ลงลึก), การเปลี่ยนแปลงโค้ดใดๆ

## แนวทางที่เลือก

แนว B (เล่าตาม path เสียง end-to-end) เป็นแกน + กรอบการแบ่งช่วงแบบเบาเพื่อจัดกลุ่ม
ตัดทฤษฎี IoT layer ที่ทำให้ซับซ้อนออก เพราะผู้ใช้ต้องการความเข้าใจง่ายที่สุด

## โครงเอกสาร

### 1. ภาพรวมการเชื่อมต่อ
- 1 ย่อหน้าอธิบายเส้นทางคำสั่งเสียง 1 คำ
- 1 figure บล็อกแนวนอน: 🎤 เสียง → [VC-02] → UART → [ESP8266] → WiFi →
  [Mosquitto] → [Backend] → SSE → [หน้าเว็บ] พร้อม label สั้นใต้แต่ละบล็อก

### 2. ไล่ทีละช่วงการเชื่อมต่อ (หัวใจของเอกสาร)
5 ช่วง ช่วงละ ~3–5 บรรทัด: เชื่อมอะไรกับอะไร / วิธี / พารามิเตอร์สำคัญ

| ช่วง | เชื่อม | วิธี/พารามิเตอร์ |
|---|---|---|
| ① เสียง → VC-02 | คนพูด → โมดูลรู้จำเสียง | ออฟไลน์ ไม่ต่อเน็ต, ตื่นด้วย "hey pudding", คำสั่งไฟ 6 แบบ |
| ② VC-02 → ESP8266 | สาย UART | SoftwareSerial 115200 8N1, cross-wire TX↔RX, packet 5 ไบต์ `[0x5a, cmd, 0x00, 0x00, checksum]` |
| ③ ESP8266 → Broker | WiFi → LAN | WiFi 2.4GHz, TCP/IP, MQTT พอร์ต 1883 |
| ④ Broker → Backend | MQTT publish/subscribe | ESP publish `dev/{id}/hb` (retained, QoS0) หลัง flip รีเลย์ → backend รับผ่าน wildcard `dev/+/hb` |
| ⑤ Backend → หน้าเว็บ | SSE | บันทึก DB + log `hb_sync` + broadcast `device_updated` ผ่าน `text/event-stream` |

### 3. ลำดับเหตุการณ์ของคำสั่งเสียง
- 1 figure sequence (ย่อจาก VOICE.md §4): User → VC-02 → ESP → Backend → UI
- อธิบายประเด็นสำคัญ: ทำไม path เสียงไม่ใช้ `dev/{id}/cmd` (ไม่มี `cmd_id`)
  แต่ใช้ heartbeat + `hb_sync` ในการ sync สถานะ — จึงไม่ต้องแก้ backend

### 4. ตารางสรุปการเชื่อมต่อทั้งหมด
ตารางเดียวรวมทุกช่วง: คอลัมน์ = ช่วง / ชนิดการเชื่อมต่อ / ระยะโดยประมาณ /
โปรโตคอล / ทิศทางข้อมูล

### 5. หมายเหตุการออกแบบ
2–3 bullet:
- ESP เป็น stateless responder — VC-02 gate wake window เองในโมดูล
- VC-02 รู้จำเสียงแบบออฟไลน์ → ความเป็นส่วนตัว + ไม่ต้องพึ่ง cloud
- Backend เป็น single source of truth — คำสั่งเสียงกับคำสั่งเว็บลงเอยที่เส้นทางเดียวกัน
